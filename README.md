# namore-landing-astro

Programmatic SEO build for `namore.app` — 11K per-name landing pages generated from the Supabase `names` table.

## Architecture (locked 2026-05-20)

- **Data source: Supabase `names` table.** Build-time fetch via the service role key. The CSV at `Documents/Projects/Namore/baby-names-production.csv` is backup-only and must not feed this build.
- **English-only in v1.** The `names` schema has `_de` / `_nl` / `_es` columns from migration 010, but the v1 page template does NOT read them. Translation data is currently weak; locale variants are explicitly out of v1 scope and only revisited if/when translation quality improves.
- **No LLM enrichment in v1.** Pages are templated from existing structured columns (`meaning`, `longevity_read`, `trend_context`, `popularity_history`, `nicknames`, `vibe_tags`, etc.). LLM-enhanced content is a v2 decision and must NOT use Claude Haiku (corrupted the dataset previously).
- **No CSV input.** `getStaticPaths()` queries Supabase directly. CSV is backup-only.
- **Per-name OG images** are generated at request time by a Vercel Edge function (separate project), not pre-rendered.
- **Static output.** Astro generates pre-rendered HTML. No client-side Supabase calls.

## Local dev

```bash
# Install
npm install

# Render against mock data (no Supabase needed)
USE_MOCK_DATA=1 npm run dev

# Build to static HTML against mock data
USE_MOCK_DATA=1 npm run build:mock

# Output appears in dist/
```

## Production build (Vercel)

Vercel project env vars required:

| Var | Value | Notes |
|---|---|---|
| `SUPABASE_URL` | `https://<project>.supabase.co` | Same as the app's `EXPO_PUBLIC_SUPABASE_URL`. |
| `SUPABASE_SERVICE_ROLE_KEY` | The `service_role` secret from Supabase Settings → API | Server-side only. Bypasses RLS. NEVER commit. |
| `BUILD_LIMIT` (optional) | e.g. `100` for the Phase 2 test batch | Omit to render all ~11K names. |

Vercel runs `npm run build` on push. Build reads from Supabase, generates 11K static HTML pages + a sitemap split into 2K-URL chunks, deploys.

## Rebuild triggers

- **Weekly schedule** (Vercel Cron) so name updates land within a week even without explicit triggers.
- **Supabase webhook** on `names` table changes (`INSERT` / `UPDATE` / `DELETE`) → Vercel deploy hook → rebuild → live within minutes.

## Routes

- `/names/[slug]` — one page per name (slug = lowercased name, URL-encoded).
- `/sitemap-index.xml` + `/sitemap-*.xml` — auto-generated by `@astrojs/sitemap`, chunked at 2K URLs each.
